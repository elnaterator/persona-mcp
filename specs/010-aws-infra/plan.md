## Summary

Stand up Terraform infrastructure-as-code for the Persona application targeting AWS us-west-2. The application runs as a Docker container on AWS Lambda (via Lambda Web Adapter — zero code changes to the app), exposed via Lambda Function URL. Neon PostgreSQL is used for the database (connection config only; Neon project managed externally). Two fully isolated environments (`dev`, `prod`) use separate Terraform directories with shared modules. Secrets are managed via SSM Parameter Store (SecureString). A new CI job validates Terraform on every PR (lint + Checkov security scan + plan for both envs) — no `apply` ever runs automatically.

## Technical Context

**Language/Version**: HCL (Terraform 1.7+)
**Primary Dependencies**: `hashicorp/aws` provider ~5.x, `hashicorp/terraform` 1.7+
**Storage**: Remote state in S3 + DynamoDB (bootstrapped manually); app uses Neon PostgreSQL (connection config via SSM)
**Testing**: `terraform fmt -check`, `terraform validate`, `terraform plan`, Checkov security scan
**Target Platform**: AWS us-west-2; Lambda (container image), ECR, SSM, CloudWatch, IAM
**Project Type**: Infrastructure-only (new `infra/` directory; no changes to Python package)
**Performance Goals**: Lambda 512 MB memory, 30s timeout; CloudWatch alarm triggers on ≥1 error in 5 min
**Constraints**: No `terraform apply` in CI; no secrets in code or state; app Dockerfile needs one added line (Lambda Web Adapter COPY)
**Scale/Scope**: 2 environments (dev, prod), 1 developer, ~10 Terraform resources per environment

## Phase 0: Research Findings

See `research.md` for full findings. Key decisions:

1. **Lambda HTTP layer**: Lambda Web Adapter + Docker container on ECR + Lambda Function URL (`auth_type = NONE`)
2. **Multi-env structure**: Separate directories (`infra/dev/`, `infra/prod/`) — not workspaces
3. **Secrets**: SSM Parameter Store (SecureString) with `lifecycle { ignore_changes = [value] }` — placeholder value set by Terraform, real value set by developer
4. **Observability**: CloudWatch Log Group (14-day retention) + error alarm (≥1 error in 5 min)
5. **CI**: Checkov (replaces deprecated tfsec), OIDC for AWS auth, matrix strategy for dev/prod plan, triggered on PR only
6. **Dockerfile addition**: `COPY --from=public.ecr.aws/awsguru/aws-lambda-adapter:0.8.4 /lambda-adapter /opt/extensions/lambda-adapter` + `ENV AWS_LWA_PORT=8000`
7. **Remote state**: S3 + DynamoDB, one backend config per environment, bootstrapped manually

## Phase 1: Design Artifacts

### Module Design: `infra/modules/lambda`

**Resources**:
- `aws_ecr_repository` — container registry per environment (`persona-{env}`)
- `aws_iam_role` — Lambda execution role (`persona-{env}-lambda-exec`)
- `aws_iam_role_policy` (2) — ECR pull permissions + SSM read + KMS decrypt
- `aws_iam_role_policy_attachment` — AWS managed `AWSLambdaBasicExecutionRole`
- `aws_lambda_function` — container image Lambda (`package_type = Image`, 512 MB, 30s timeout, `architecture = x86_64`)
- `aws_lambda_function_url` — public HTTPS endpoint (`authorization_type = NONE`)

**Key implementation notes**:
- `image_uri = "${aws_ecr_repository.app.repository_url}:${var.image_tag}"` — references ECR image
- Lambda env var `AWS_LWA_PORT = "8000"` passed as environment variable (overrides adapter default)
- ECR `scan_on_push = true` for free vulnerability scanning
- Lambda Function URL CORS configured to allow all origins (app handles auth via Clerk)

### Module Design: `infra/modules/observability`

**Resources**:
- `aws_cloudwatch_log_group` — `/aws/lambda/{function_name}`, retention 14 days (dev: 7 days)
- `aws_cloudwatch_metric_alarm` — `Errors` metric, `Sum` statistic, period 300s, threshold 1, `treat_missing_data = notBreaching`

**Key implementation notes**:
- Log group must be created **before** Lambda function to prevent CloudWatch from creating it with no retention policy
- `depends_on` relationship: Lambda module depends on observability module for log group creation

### Environment Root Design

Each environment root (`infra/dev/main.tf`, `infra/prod/main.tf`) wires the modules together and defines SSM parameters:

```
module "lambda" { source = "../modules/lambda"; ... }
module "observability" { source = "../modules/observability"; lambda_function_name = module.lambda.lambda_function_name; ... }

resource "aws_ssm_parameter" "database_url" { name = "/persona/{env}/database_url"; ... lifecycle { ignore_changes = [value] } }
resource "aws_ssm_parameter" "clerk_secret_key" { ... }
resource "aws_ssm_parameter" "clerk_publishable_key" { ... }
```

### Dependency Management Policy

`uv` is assumed to be installed (it is a documented prerequisite). All Python tooling is managed through `uv`:

- **Managed packages** (Python deps declared in `pyproject.toml`): installed via `uv sync` as part of `make setup`
- **Run-once CLI tools** (e.g. `checkov`): invoked via `uvx <tool>` — downloaded and cached automatically by `uv` on first use, no explicit install step, no global pip install, fully cross-platform

`make setup` (root Makefile) runs `uv sync` (backend) + `npm ci` (frontend) after a fresh clone. It does not need to pre-install `checkov`; `uvx checkov` handles that lazily.

### CI Job Design: `.github/workflows/terraform-ci.yml`

**Job 1: `terraform-lint-scan`** (no AWS credentials needed)
- `actions/checkout@v4`
- `hashicorp/setup-terraform@v3` (version: ~1.7)
- `terraform fmt -check -recursive` (runs from `infra/`)
- `astral-sh/setup-uv@v4` — installs `uv` (and `uvx`) on the runner
- `uvx checkov -d infra/ --quiet --compact` — no `pip install` step required

**Job 2: `terraform-plan`** (matrix: dev, prod; needs: terraform-validate)
- `actions/checkout@v4`
- `hashicorp/setup-terraform@v3`
- `aws-actions/configure-aws-credentials@v4` (OIDC role: `github-actions-terraform-{env}`)
- `terraform -chdir=infra/${{ matrix.environment }} init`
- `terraform -chdir=infra/${{ matrix.environment }} plan -no-color`

**Trigger**: `on: pull_request` with `paths: ['infra/**']`
**No apply**: Zero `terraform apply` commands in any job or workflow.

### Dockerfile Change

In the runtime stage (Stage 3), before `EXPOSE 8000`:

```dockerfile
# Lambda Web Adapter — enables Lambda to proxy events to the app's HTTP server
COPY --from=public.ecr.aws/awsguru/aws-lambda-adapter:0.8.4 /lambda-adapter /opt/extensions/lambda-adapter
ENV AWS_LWA_PORT=8000
```

This is a one-line addition that does not affect local Docker Compose usage (`AWS_LWA_PORT` is benign when not running on Lambda).

### Makefile Additions (root `Makefile`)

```makefile
setup:          # uv sync (backend) + npm ci (frontend) — run once after fresh clone
tf-lint:        # terraform fmt -check -recursive in infra/
tf-check:       # tf-lint + uvx checkov scan (no pip install required)
```

`make check` depends on `tf-lint` so formatting violations are caught before commit.

## Key Constraints (Implementation Guard Rails)

1. **Never add `terraform apply` to any CI job** — the spec explicitly prohibits this (FR-010, SC-005)
2. **Never hardcode secret values in `.tf` or `.tfvars` files** — SSM placeholder pattern only (FR-012)
3. **`infra/dev/` and `infra/prod/` are fully independent** — no shared state, no cross-references between environment roots (FR-003)
4. **Dockerfile change is additive only** — the Lambda Web Adapter COPY line does not break local Docker Compose operation (AWS_LWA_PORT env var is ignored outside Lambda)
5. **Bootstrap resources (S3 + DynamoDB) are NOT Terraform-managed** — documented in runbook, never added to any `.tf` file
6. **Lambda module depends on observability module** — log group must be created before Lambda function to avoid AWS auto-creating an unretained log group
